Estructura general de AISync AISync organiza el trabajo asistido por IA mediante una jerarquía operativa clara: **Project → Team → Workspace → AI Agent Session** Esta estructura permite que el sistema no funcione como una colección suelta de chats, sino como una arquitectura de trabajo trazable, organizada y recuperable. --- ## 2. Project Un **Project** es el contenedor principal de trabajo. Representa una línea de trabajo, caso, cliente, iniciativa o problema mayor. Dentro de un Project se agrupan todos los Teams, Workspaces, archivos de contexto, eventos, checkpoints y documentación relacionados con ese frente. El Project no es una conversación. El Project no es un Team. El Project es el marco superior donde vive una parte organizada del trabajo. Ejemplos: * Proyecto A — Cliente inmobiliario * Proyecto B — Auditoría interna * Proyecto C — Desarrollo de producto * Proyecto D — Investigación estratégica Cada Project debe poder contener uno o varios Teams. --- ## 3. Team Un **Team** es una unidad interna de coordinación dentro de un Project. Sirve para dividir el trabajo por función, especialidad, etapa o frente operativo. Ejemplos: * Strategy Team * Legal Review Team * UX Team * Research Team * Commercial Team * Documentation Team Cada Team puede tener su propia configuración, sus propios agentes, sus propios Workspaces, sus propios archivos de contexto y su propia trazabilidad. En AISync, un Team no debe leerse como una simple tarjeta visual. Es una unidad real de trabajo dentro del sistema. --- ## 4. Workspace Un **Workspace** es la superficie operativa persistente donde trabaja un Team. Mientras el Team representa la unidad de coordinación, el Workspace representa el lugar concreto donde ocurre el trabajo. Un Team puede tener uno o varios Workspaces, según la complejidad del trabajo. Ejemplo: **Project A** * Team: Strategy * Workspace: Main Strategy Workspace * Workspace: Client Brief Workspace * Workspace: Decision Review Workspace El Workspace conserva continuidad operativa. No debe confundirse con una sesión descartable. --- ## 5. AI Agent Session Una **AI Agent Session** es la instancia activa de trabajo de un agente dentro de un Workspace. Es el ciclo vivo donde el usuario interactúa con la IA, se producen respuestas, se cargan archivos, se genera contexto y se crean eventos relevantes. La sesión puede refrescarse, pausarse, retomarse o producir checkpoints, pero siempre debe estar asociada a un Workspace, un Team y un Project. La lectura correcta es: **el Project contiene el trabajo; el Team lo coordina; el Workspace lo aloja; la AI Agent Session lo ejecuta.** --- # 6. Función de Team Map **Team Map** debe ser la vista estructural que permite entender cómo está organizado el trabajo dentro de AISync. No debe ser solo un mapa decorativo. Debe permitir ver: * qué Projects existen; * qué Teams pertenecen a cada Project; * qué Teams están subordinados o conectados; * qué Teams tienen Workspaces activos; * qué Team está operando actualmente; * cómo crece la estructura del sistema; * cómo se relacionan los distintos frentes de trabajo. Team Map debe mostrar la elasticidad organizacional de AISync: la capacidad de crear, subdividir, conectar y reorganizar Teams sin perder claridad ni trazabilidad. --- # 7. Representación visual de Projects en Team Map En el rediseño de Team Map, los **Projects deben diferenciarse visualmente**. No alcanza con mostrar todos los Teams mezclados en un único plano. Los Teams contenidos dentro de un mismo Project deben estar **enmarcados, agrupados o contenidos visualmente** dentro de una zona clara de Project. Esto puede resolverse mediante: * marcos por Project; * contenedores visuales; * zonas o clusters; * fondos suaves por Project; * bordes de grupo; * encabezados de Project; * agrupación espacial clara. El objetivo es que el usuario pueda entender de inmediato: **estos Teams pertenecen al Project A** **estos otros Teams pertenecen al Project B** **este Team está dentro de este Project, no suelto en el sistema** El Project debe funcionar visualmente como un contenedor superior. --- # 8. Color de Teams Cada **Team debe tener su propio color**. Ese color debe servir para identificar rápidamente al Team dentro del mapa, dentro del árbol y dentro de futuras superficies operativas relacionadas. Regla recomendada: **cada Team tiene un color propio, pero los colores deben convivir dentro de una lógica visual por Project.** Esto significa: * el Project puede tener un marco, fondo o acento propio; * cada Team dentro de ese Project debe tener su propio color identificable; * los subteams pueden derivar visualmente del Team padre; * los workers o nodos internos deben usar una versión más liviana del color del Team; * el color no debe usarse para indicar SAT/MAT, estado o prioridad principal. SAT/MAT, estado, alertas o conexión deben mostrarse con badges, bordes, iconos o etiquetas, no compitiendo con el color estructural del Team. --- # 9. Regla visual recomendada La lógica visual debería ser: **Project = marco / contenedor / zona visual** **Team = color propio** **Subteam = variación derivada del Team padre** **Worker = color heredado con menor intensidad** **Estado = badge o borde secundario** **SAT/MAT = badge textual o etiqueta estructural** Esto evita que el mapa se vuelva caótico. El color debe ayudar a reconocer pertenencia y ubicación, no reemplazar la jerarquía. --- # 10. Cómo debería leerse Team Map El usuario debe poder mirar Team Map y entender tres cosas sin esfuerzo: ## A. Qué Projects existen Cada Project debe aparecer como un bloque, zona o contenedor visual. ## B. Qué Teams pertenecen a cada Project Los Teams deben estar claramente dentro del marco de su Project. ## C. Cómo se organiza el trabajo dentro de cada Project Debe verse si hay Teams principales, subteams, conexiones, Workspaces o ramas operativas. La lectura ideal es: **Project A contiene estos Teams. Cada Team tiene su color. Cada Team puede abrir sus Workspaces. Cada Workspace puede contener AI Agent Sessions. El sistema puede crecer sin perder estructura.** --- # 11. Criterio para rediseño de Team Map El rediseño de Team Map debe respetar estos principios: 1. **No mezclar Projects visualmente.** Los Projects deben tener contención visual clara. 2. **No usar un único plano indiferenciado.** Si todos los Teams aparecen flotando juntos, se pierde la jerarquía. 3. **Cada Team debe tener identidad visual propia.** El color del Team ayuda a reconocerlo en distintas vistas. 4. **El Project debe verse como agrupador superior.** Puede tener marco, fondo, etiqueta, header o zona propia. 5. **El color no debe intentar explicar todo.** Proyecto, Team, estado, tipo de team y conexión deben tener códigos visuales separados. 6. **El mapa debe escalar.** Debe funcionar con pocos Teams y también con varios Projects y múltiples Teams por Project. 7. **Team Map debe ser operativo.** No debe ser solo una imagen: debe permitir abrir, editar, navegar, entender jerarquía y orientar el trabajo. --- # 12.
